--------------------------------------------------------------------------------
0. CONTEXT YOU NEED BEFORE TOUCHING ANYTHING
--------------------------------------------------------------------------------

WHAT THIS REPO IS
  An AOT compiler from Elixir (.beam bytecode, via OTP's :beam_disasm) to
  WebAssembly GC, plus a JSPI-based process runtime, targeting Cloudflare
  Workers / Durable Objects. The compiler is ONE file:
      compiler/beam2wasm.exs        (~3,711 lines, Elixir script)
  The process runtime is ONE file:
      runtime/scheduler.mjs         (~148 lines, Node ESM)
  Everything else is harnesses, demos, spikes, and docs.

TOOLCHAIN (exact versions matter)
  - Elixir + OTP (recent stable; the harnesses load OTP's own :gen_server etc.)
  - Node.js v24.16.0 — JSPI is experimental and version-sensitive. All Wasm
    runs need:  node --experimental-wasm-jspi
  - Binaryen wasm-as version_130 (compiles .wat -> .wasm; needs -all flag)
  - workerd (only for the durable-object demos)
  See BUILD.md for install steps. NOTE: the harnesses currently default the
  node path to /Users/ivar/.nvm/... — override with:  export NODE=$(which node)
  (Fixing that default is TODO 2.)

THE SAFETY NET — run these after EVERY change, no exceptions
  All three are differential tests: the same Elixir source runs on WasmGC and
  on the real Elixir VM, and outputs must match bit-exactly. They are the
  reason this project can claim correctness. A change is done only when all
  three are green.

    cd conformance && elixir run.exs       # 131 cases, expect 131/131 (100.0%)
    cd fuzz        && elixir run.exs       # 33 cases,  expect 33/33
    cd gaps        && elixir run.exs       # 20 programs x 8 seeds; expect
                                           # 8 "provably correct", 0 LIES
                                           # (many TRAP/STUB results are
                                           # expected — those are known gaps,
                                           # not regressions. A regression =
                                           # a previously-passing program now
                                           # failing, or any LIE.)

  Filter forms: `elixir run.exs binaries` (conformance, category substring),
  `elixir run.exs p14` (gaps, program filter).

HOW TO COMPILE ONE PROGRAM BY HAND (for debugging)
    cd compiler
    elixirc examples/mergesort.ex                      # -> Elixir.Sort.beam
    elixir beam2wasm.exs Elixir.Sort.beam > sort.wat
    wasm-as sort.wat -o sort.wasm -all
    node --experimental-wasm-jspi examples/runsort.mjs sort.wasm

COMPILER ENV FLAGS (read compiler/beam2wasm.exs lines ~40-60)
  BIGNUM=0   opt OUT of exact integers (wrapping i31 mode; experiments only —
             exact/tiered integers are the DEFAULT, despite what old docs say)
  STUB=1     unknown opcodes/functions compile to traps + a "; STUB" comment
             in the WAT instead of failing the build (the gaps harness uses
             this to measure coverage)
  REDS=<n>   inject reduction-count preemption checks at function entries
  EXPORTS=.. control which functions are exported

0.5 FALSE ALARMS — THINGS THAT LOOK LIKE BUGS BUT ARE NOT. DO NOT "FIX" THESE.
--------------------------------------------------------------------------------

(Both were flagged during review and then disproven against the source. Left
here so the next reviewer doesn't burn a day on them.)

A) "Bignum equality is broken / $term_compare has no $big case."
   FALSE. compiler/beam2wasm.exs:1882-1885 — rank 0 (numbers) dispatches to
   $int_cmp when bignum mode is on, and $int_cmp (see ~line 816) calls the
   host import big.cmp ($bigint_cmp, imported at ~line 683) for boxed values.
   Equality of two distinct-but-equal $big structs is value-correct via that
   path. The i31/i31 fast path at lines 1864-1866 only fires when BOTH
   operands are i31, so it never short-circuits a mixed i31/$big compare.
   Covered by conformance category "bignum" and the fuzz ledger.

B) "Integer arithmetic silently truncates to i32 by default."
   FALSE. Exact integers are the default: beam2wasm.exs:53-55,
   `bignum = System.get_env("BIGNUM") != "0"`. Wrapping arithmetic only
   happens if you explicitly set BIGNUM=0, which the comment marks as
   experiments-only. (Old docs describing BIGNUM as "opt-in" are stale —
   that's TODO 1.)

TODO 4 — SCHEDULER: KILL-BY-UNWIND, FAIRNESS, RECEIVE TIMEOUTS (~2-4 days)
--------------------------------------------------------------------------------

WHY: ARCHITECTURE.md section 9 mandates "terminate processes by UNWINDING,
never by dropping references" — because an abandoned suspended JSPI stack is
engine-rooted and LEAKS (~41MB/gen measured in
spikes/01-jspi-economics/spikeB.mjs, which also proved the reject-the-
parked-promise kill path works and plateaus). The real scheduler
(runtime/scheduler.mjs) does NOT implement that kill path: a process today
only ends by returning or throwing on its own. This is the one place the
architecture document promises something the shipped runtime doesn't do.
ROADMAP also calls scheduler fairness "the single most important remaining
runtime component".

CURRENT STATE (runtime/scheduler.mjs, 148 lines — read all of it first,
it's small and dense):
  - Three queues: toStart (new spawns), toResume (got a message while
    waiting), toReady (preempted/yielded). Dispatch loop at lines 138-140
    drains toStart FULLY before toResume before toReady -> a spawn-heavy
    process starves everyone else.
  - Parking: proc.recv_wait and sched.yield are WebAssembly.Suspending
    imports that create a Promise and stash its `resolve` on the process
    record (p.resolve). Resuming = calling p.resolve().
  - NO p.reject is stashed -> nothing can unwind a parked stack.
  - finish(pid, ok) at lines ~91-117 handles links (trap_exit ->
    {:EXIT,...} message; abnormal + no trap -> recursively finish the
    linked proc) and monitors (:DOWN messages). NOTE: for a linked process
    that is parked, finish() currently just marks it dead WITHOUT unwinding
    its stack — that is exactly the leak path spike B warned about. Your
    kill mechanism must be used here too, not just for explicit kills.
  - No receive-after: proc.recv_wait parks forever; Elixir
    `receive ... after` is not compilable today.

TASK:
4.1  Kill path. Stash BOTH resolve and reject when parking (recv_wait and
     yield). Add kill(pid, reasonRef): if the target is parked, call
     p.reject(new ProcKill(reasonRef)) which unwinds the Wasm stack through
     JSPI; the existing run.then(..., () => finish(pid, false)) at line
     ~122 then fires finish() normally. If the target is queued-but-not-
     parked (toStart/toReady), remove it from the queue and finish it
     directly (no stack exists yet / it's between suspensions). Route the
     existing abnormal-link-propagation branch in finish() through kill()
     so linked parked processes are unwound, not abandoned.
     Make sure finish() is idempotent (guard on p.dead) — kill during
     cascade can re-enter it.
4.2  Expose it to Elixir: the compiler already lowers
     Process.exit/2-shaped BIFs to a "proc" import (check the proc import
     list in scheduler.mjs lines 30-58 — `exit` exists for self-exit;
     extend for exit(pid, reason)). Cross-check what the compiler emits:
     grep beam2wasm.exs for `"exit"` / spawn_opt handling around lines
     2400-2510 before assuming the import name.
4.3  Fairness. Replace strict queue priority with interleaving — simplest
     correct option: one combined FIFO of {kind, pid} entries, or
     round-robin one item per queue per loop turn. Don't overbuild;
     BEAM-style per-process accounting can come later. The invariant that
     matters: a continuously-spawning main process must not starve
     toResume/toReady forever.
4.4  receive-after timers. Parking with a timeout: in recv_wait, if the
     process has a pending `after` deadline, use setTimeout to resolve the
     park with a "timed out" marker the compiled code can branch on. This
     needs compiler cooperation (the wait_timeout BEAM opcode — check
     whether beam2wasm.exs handles wait_timeout at all; if not, that's the
     other half of this task and it's a compiler change: grep for
     `:wait_timeout` / `:wait` in the emit case around lines 2067-2637).
     If the compiler half is too big, ship 4.1-4.3 alone — they're
     independent.

TESTS (add to conformance — this is mandatory, the suite is the contract):
  - kill of a parked process: parent spawns child that parks in receive,
    parent kills it, monitors confirm :DOWN with the right reason.
  - abnormal exit cascading through a link chain of 3+ parked processes.
  - spawn-storm fairness: main loops spawning trivial children while a
    sibling pingpongs messages; assert the sibling makes progress (this
    one may need to be a runtime/ unit test rather than a differential
    test, since the VM oracle trivially passes it).
  - if 4.4 lands: receive-after-0 (mailbox poll) and receive-after-100.
MEMORY CHECK: rerun the spike B abandon-vs-kill measurement shape against

processes, kill all, force GC (node --expose-gc), assert heap returns to
baseline.

DO THE TWO SMALL FIXES FIRST (independent, low-risk):

5.1  STUB operand fallback emits silent wrong code.
     beam2wasm.exs:2743-2745:
       defp operand(o) do
         if Process.get(:stub), do: "(ref.null none)", else: raise(...)
       end
     Under STUB=1 an UNRECOGNIZED OPERAND becomes nil. Unlike opcode-level
     stubs (which compile to traps and are honestly counted in the "STUBS:"
     meter), this can produce a wrong VALUE that flows onward — the only
     known path to a silent wrong answer in STUB mode, which would
     undermine the gaps harness's "0 lies" guarantee. FIX: always raise, or
     emit `(unreachable)`-guarded code; never a usable nil. Then run the
     gaps harness — if nothing regresses, no program was relying on it
     (expected: the catch-all is for operand SHAPES we've never seen).

5.2  Error context on unhandled constructs. beam2wasm.exs:2364
     (`raise "bs_match cmd: ..."`) and :2635
     (`raise "unhandled in #{name}/#{arity}: ..."`). 2635 has fn/arity;

     a hint ("set STUB=1 to compile this to a trap and continue").
